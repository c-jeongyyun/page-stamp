## 1. 빌더: esbuild

**선택 이유**
- 빠른 빌드 속도

**Trade-off**
- HMR(Hot Module Replacement) 불가

**해결 방안**
- `--watch` 옵션 + Figma 데스크탑 앱의 **Auto-reload plugin** 옵션 활성화로 대응
  - 저장 → 자동 빌드 → 플러그인 자동 재시작 흐름 구성
  - 재빌드 시 상태 초기화는 불가피하나, 이 플러그인은 전역으로 유지해야 하는 상태가 없고 상태 흐름 자체가 복잡하지 않아 DX 저하에 미치는 영향은 미미할 것으로 판단

## 2. 프레임워크: Preact + preact/compat

**선택 이유**
- Figma 플러그인 UI는 iframe으로 렌더링되는 구조로, 번들 크기가 크면 플러그인 첫 화면 로드가 느려질 것으로 판단
- React 대비 번들 크기가 작은 Preact 선택
- `preact/compat` 사용으로 React와 동일한 개발 경험 유지

**Vanilla 미선택 이유**
- 유지보수 관점에서 직관적이지 않은 구조가 될 것으로 판단

**preact/compat 호환성 관련 유의사항**
- preact/compat은 React 생태계 라이브러리 중 복잡한 ref 패턴이나 concurrent 기능에 의존하는 것들과 호환되지 않을 수 있음
- 단, 이 플러그인은 기능 형태상 그러한 고도화된 기능을 사용할 일이 없을 것으로 예상되어 문제 소지는 낮다고 판단
- 현재는 vibe coding을 통한 UI 요소 자체 구축 예정이나, 추후 외부 UI 라이브러리 사용하게 될 시 유념 필요
